## 1. Goal

Build the server half of **SyncWire** — a system where an Android app installed on Phone A captures its SMS and system notifications, forwards them to this server, and the server relays them in real time to Phone B (another Android device with the same app) or to a web dashboard.

A user with one or more "source" phones can have notifications mirror to one or more "viewer" devices, paired by short code.

**Non-goals (v1):** cross-platform push (iOS), end-to-end encryption of notification content, group/team accounts, billing, analytics dashboard.

---

## 2. Why MQTT, not WebSocket

Original code used Socket.IO. We're moving to MQTT via EMQX because:

| Concern | WebSocket (current code) | MQTT (proposed) |
|---|---|---|
| Routing to specific recipient | hand-rolled per-client socket IDs | topic-based — broker handles fan-out |
| Presence / "is device online?" | custom heartbeat + DB column | **LWT + retained message**, broker-native |
| Offline message catch-up | hand-rolled store + cursor REST API | QoS 1 + persistent session, plus DB for >session-expiry |
| Auth at the connection | none | JWT validated by EMQX on connect |
| Per-device ACL | none | declarative topic rules with placeholders (`syncwire/inbox/${clientid}`) |
| Scaling beyond one node | sticky sessions, custom state | broker cluster, shared subscriptions |
| Multiple concurrent servers processing the same stream | hard | shared subscriptions (`$share/group/topic`) |
| Battery-friendly mobile client | poll-based | standard MQTT keepalive, well-supported on Android |

**Two planes:**
- **Data plane (MQTT):** notifications flow over MQTT topics. Phone A publishes, broker routes to phone B's inbox.
- **Control plane (REST/HTTP):** auth, pairing, device management, history query, health checks.

A **single NestJS process** hosts the REST control plane AND subscribes to a `syncwire/relay/+` topic as an MQTT client to do persistence + fan-out logic. (We could let the broker do pure ACL-based fan-out with no app code in the hot path, but the bridge gives us filtering, multi-receiver fan-out, audit, and rate-limiting. It's the right tradeoff for v1.)

## 4. Topic design

All topics live under the `syncwire/` root namespace. Topic placeholders use `${clientid}` and `${username}`.

### Data plane

| Topic | Direction | Who publishes | Who subscribes | QoS | Retain | Purpose |
|---|---|---|---|---|---|---|
| `syncwire/relay/${sourceDeviceId}` | Phone A → server | source device | NestJS bridge | 1 | false | raw notification from a source device |
| `syncwire/inbox/${viewerDeviceId}` | server → Phone B | NestJS bridge | viewer device | 1 | false | notifications intended for a specific viewer |
| `syncwire/status/${deviceId}` | device → server, server mirrors | each device | NestJS bridge, any paired device (read-only) | 0 | true | presence: `{"status":"online"\|"offline","at":<ts>}` |
| `syncwire/cmd/${deviceId}` | server → device | NestJS bridge | a specific device | 1 | false | future: revoke pairing, force reauth, etc. |

### Internal (bridge-only)

| Topic | Purpose |
|---|---|
| `syncwire/bridge/audit` | NestJS bridge writes every relayed message here for audit/replay — not subscribed by anyone in v1, but easy to add |
| `$SYS/#` | EMQX internal — for monitoring, never publish |

### Why this shape

- **Source device id is in the topic path, not just the payload.** This means EMQX ACLs can grant "device A can publish to `syncwire/relay/A`" without inspecting the payload — defense in depth against a compromised client trying to spoof another device's source.
- **Viewer inbox is also in the topic path.** Same reason: a device can only subscribe to its own inbox. The bridge can republish to any inbox (it's a privileged server-side publisher).
- **Status uses retain=true** so a late subscriber (e.g., a web dashboard) immediately sees the current state without waiting for a heartbeat.
- **Status uses QoS 0** because presence is best-effort and we don't want offline queue buildup.
- **Data uses QoS 1 + retain=false** because we want at-least-once to the broker, and we don't want stale "last notification" sitting in the broker — history is the database's job.

### Wildcard subscriptions

- The bridge subscribes to `syncwire/relay/+` (single-level) to receive from all sources.
- A future "admin dashboard" can subscribe to `syncwire/status/+` (single-level) to watch all device presence without subscribing to notification content.

## 5. Auth model

### Users and devices

- A **user** owns one or more **devices**.
- A **device** is one physical phone (or browser tab, identified by a per-install UUID).
- A **pairing** links two users (or one user across two devices, or two distinct users) so that notifications from device A flow to device B.

### Tokens

| Token | Lifetime | Storage | Used for |
|---|---|---|---|
| **Access JWT** (HS256) | 1 hour | device memory | REST API (Authorization: Bearer), MQTT CONNECT password |
| **Refresh token** (opaque random) | 90 days, rotating | device secure storage (Android Keystore / httpOnly cookie for web) | exchange for new access JWT |
| **Pairing code** (6 digits) | 10 minutes, single use | shown on receiver, entered on sender | bootstrap a pairing |

JWT claims:
```json
{
  "sub": "user_<uuid>",
  "devices": ["dev_<uuid>"],
  "iat": 1700000000,
  "exp": 1700003600,
  "scope": ["relay:publish", "inbox:subscribe", "status:read"]
}
```

EMQX is configured with a **JWT authenticator** (secret in env, shared with NestJS). On every CONNECT, EMQX validates the JWT signature and extracts the claims. The **ACL file** then maps `username` and `clientid` to allowed topics:

```hcl
# /etc/emqx/acl.conf (simplified, real file below)
{allow, {username, {re, "^user_"}}, all, ["syncwire/inbox/${clientid}"]}.
{allow, {clientid, {re, "^dev_"}},   publish, ["syncwire/relay/${clientid}"]}.
{allow, {username, {re, "^user_"}},  all, ["syncwire/status/${clientid}", "syncwire/cmd/${clientid}"]}.
{deny, all, all, ["#"]}.
```

> The "user" clientid pattern (`dev_<userUuid>_<deviceUuid>`) lets the ACL match without a database lookup. Fan-out across users happens in the bridge, not the broker.

### Pairing flow

```
Phone B (viewer)                Server (NestJS)                Phone A (source)
       │                              │                              │
       │  POST /api/pairings/start    │                              │
       │  (auth: JWT)                 │                              │
       │─────────────────────────────►│                              │
       │                              │  generate 6-digit code       │
       │                              │  store: code, expiresAt,     │
       │                              │    initiatorDeviceId         │
       │  201 { code, expiresAt,      │                              │
       │       pairingId }            │                              │
       │◄─────────────────────────────│                              │
       │                              │                              │
       │  (display code on screen)    │                              │
       │                              │                              │
       │                              │  POST /api/pairings/claim    │
       │                              │  { code }   (auth: JWT)      │
       │                              │◄─────────────────────────────│
       │                              │                              │
       │                              │  resolve code → initiatorId  │
       │                              │  insert pairing row          │
       │                              │  publish syncwire/cmd/<B>:   │
       │                              │    pairing-claimed           │
       │                              │  publish syncwire/cmd/<A>:   │
       │                              │    pairing-claimed           │
       │  WS push: pairing-claimed    │                              │
       │◄─────────────────────────────│                              │
       │                              │  201 { pairingId }           │
       │                              │─────────────────────────────►│
       │                              │                              │
       │  now both subscribe          │                              │  now subscribes inbox
       │  syncwire/inbox/B            │                              │  syncwire/inbox/A (if A
       │                              │                              │  is also a viewer — usually
       │                              │                              │  source phones don't have
       │                              │                              │  their own inbox mirror)
```

Edge cases on pairing:
- Code reuse: server marks the code used atomically (DB unique constraint + check expiresAt in the same transaction).
- Code expiry: client shows countdown, refetches via `GET /api/pairings/active` if it expires.
- Either side cancels: `DELETE /api/pairings/:id` works for either participant.
- Self-pairing (same user, two devices): allowed, just links two device IDs of the same user.
- Same source already paired to same viewer: returns the existing pairing, no duplicate.

## 6. Data model (Postgres)

```sql
-- users
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         TEXT UNIQUE,
  display_name  TEXT NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- devices (one physical install of the app)
CREATE TABLE devices (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name            TEXT NOT NULL,                -- e.g. "Mohsin's Pixel 8"
  platform        TEXT NOT NULL CHECK (platform IN ('android', 'web')),
  push_token      TEXT,                          -- FCM token for Android (future push fallback)
  last_seen_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at      TIMESTAMPTZ                    -- soft-revoke a lost device
);
CREATE INDEX devices_user_id_idx ON devices(user_id);

-- refresh tokens (rotating)
CREATE TABLE refresh_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  device_id   UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
  token_hash  TEXT NOT NULL UNIQUE,             -- SHA-256 of the opaque token
  expires_at  TIMESTAMPTZ NOT NULL,
  revoked_at  TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX refresh_tokens_user_id_idx ON refresh_tokens(user_id);

-- pairings (one row per direction A→B; A is the "source-of", B is the "viewer-of")
-- We store direction explicitly so one user can be source for some viewers and
-- viewer for other sources.
CREATE TABLE pairings (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  source_device_id  UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
  viewer_user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  viewer_device_id  UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
  status        TEXT NOT NULL CHECK (status IN ('active', 'revoked')),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at    TIMESTAMPTZ,
  UNIQUE (source_device_id, viewer_device_id)  -- prevent duplicates
);
CREATE INDEX pairings_source_idx ON pairings(source_device_id) WHERE status = 'active';
CREATE INDEX pairings_viewer_idx ON pairings(viewer_device_id) WHERE status = 'active';

-- notification log (every notification, retained 30 days by default)
CREATE TABLE notifications (
  id              TEXT PRIMARY KEY,             -- client-supplied dedup id
  source_user_id  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  source_device_id UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
  source_type     TEXT NOT NULL CHECK (source_type IN ('SMS', 'NOTIFICATION')),
  sender          TEXT NOT NULL,                -- SMS sender or app label
  content         TEXT NOT NULL,
  device_timestamp BIGINT NOT NULL,             -- epoch ms from the source device
  package_name    TEXT NOT NULL,
  server_received_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  delivered_to    UUID[] NOT NULL DEFAULT '{}'  -- viewer device ids that have ACK'd
);
CREATE INDEX notifications_source_device_received_idx
  ON notifications(source_device_id, server_received_at DESC);

-- one-time pairing codes (short-lived, never leaves the request flow)
CREATE TABLE pairing_codes (
  code              TEXT PRIMARY KEY,           -- 6 digits
  initiator_user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  initiator_device_id UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
  expires_at        TIMESTAMPTZ NOT NULL,
  used_at           TIMESTAMPTZ
);
```

Migrations managed by **Prisma**.

## 7. REST API surface

All routes under `/api`. All require `Authorization: Bearer <jwt>` except `/auth/*` and `/health`.

| Method | Path | Purpose | Notes |
|---|---|---|---|
| `GET`  | `/health` | liveness + dependency checks | public, no auth. checks DB, EMQX connection |
| `POST` | `/auth/register` | create user + first device | returns access JWT + refresh token |
| `POST` | `/auth/login` | email + password → JWT | bcrypt-hashed password stored on `users.password_hash` (add column in v1) |
| `POST` | `/auth/refresh` | rotate refresh token | returns new access JWT + new refresh |
| `POST` | `/auth/logout` | revoke refresh token | |
| `GET`  | `/devices` | list my devices | includes `last_seen_at` |
| `POST` | `/devices` | register a new device (multi-device) | name, platform, push_token |
| `DELETE` | `/devices/:id` | revoke a device | cascades refresh tokens, doesn't delete notifications |
| `POST` | `/pairings/start` | create a pairing code | returns 6-digit code, 10-min expiry, pairingId |
| `POST` | `/pairings/claim` | redeem a code | body: `{ code }` |
| `GET`  | `/pairings` | list active pairings | both as-source and as-viewer |
| `DELETE` | `/pairings/:id` | revoke a pairing | either side can revoke |
| `GET`  | `/notifications?since=<id>&limit=<n>&deviceId=<id>` | history fetch | for catch-up after offline period |
| `POST` | `/notifications` | (optional) HTTP fallback to ingest | mirrors the MQTT ingest path; MQTT is primary |
| `POST` | `/notifications/:id/ack` | mark a notification delivered (viewer ACK) | updates `notifications.delivered_to` |

### Why keep `POST /notifications` over HTTP if MQTT is the primary ingest path?

Three reasons:
1. **Easier testing** — `curl` works without an MQTT client.
2. **Fallback for clients behind firewalls** that block outbound MQTT (rare, but happens on corporate networks).
3. **Simpler first iteration on the Android side** — the app can POST over HTTPS first, then migrate to MQTT once the bridge is solid. Same DTO, same service code, called from either a REST controller or an MQTT message handler.

## 8. MQTT message contracts

### Inbound: source → server (published by phone A)

Topic: `syncwire/relay/<sourceDeviceId>`
QoS: 1, retain: false

```json
{
  "id": "ntf_<uuid>",
  "sourceType": "SMS",
  "sender": "+8801XXXXXXXXX",
  "content": "Your OTP is 123456",
  "timestamp": 1718540000000,
  "packageName": "com.google.android.apps.messaging",
  "schemaVersion": 1
}
```

`schemaVersion` is reserved for future payload changes. v1 ignores it but rejects unknown major versions.

### Outbound: server → viewer (published by the bridge)

Topic: `syncwire/inbox/<viewerDeviceId>`
QoS: 1, retain: false

```json
{
  "id": "ntf_<uuid>",                    // matches source id, lets receiver dedup
  "source": {
    "userId": "usr_<uuid>",
    "deviceId": "dev_<uuid>",
    "deviceName": "Mohsin's Pixel 8"
  },
  "sourceType": "SMS",
  "sender": "+8801XXXXXXXXX",
  "content": "Your OTP is 123456",
  "timestamp": 1718540000000,
  "serverReceivedAt": "2026-06-16T10:00:00.123Z",
  "packageName": "com.google.android.apps.messaging",
  "schemaVersion": 1
}
```

`source` block added by the bridge so the viewer can display "From: Mohsin's Pixel 8" — useful when one viewer is paired with multiple sources.

### Status (retained)

Topic: `syncwire/status/<deviceId>`
QoS: 0, retain: true

```json
{ "status": "online",  "at": "2026-06-16T10:00:00.000Z" }
{ "status": "offline", "at": "2026-06-16T10:05:00.000Z" }
```

LWT configured at CONNECT time: `lastWillTopic=syncwire/status/<deviceId>`, `lastWillMessage={"status":"offline"}`, `lastWillRetain=true`, `lastWillQoS=0`. On normal connect, the device publishes `{"status":"online"}` with `retain=true` to the same topic.

## 9. The bridge service (NestJS side of the data plane)

A single NestJS provider that:
1. On boot, opens an MQTT client connection to EMQX using a service-account JWT (a special `bridge@system` user with all-topic pub/sub rights).
2. Subscribes to `syncwire/relay/+` and `syncwire/status/+`.
3. On each `relay/<id>` message:
   - Validates the payload against `CreateNotificationDto`.
   - Looks up the source device, asserts it's not revoked.
   - Persists to `notifications` table.
   - Looks up all active pairings where `source_device_id = <id>`.
   - For each viewer, publishes to `syncwire/inbox/<viewerDeviceId>` (QoS 1).
   - Emits an audit event.
4. On each `status/<id>` message:
   - Updates `devices.last_seen_at` (debounced — at most once per 30s per device).
   - Emits a presence event (in case the web UI wants to subscribe to a local WebSocket for live updates — separate namespace, but a future phase).

### Resilience

- **Outbox pattern for publish reliability.** Persist first, then publish. If the bridge crashes after persistence but before publish, a periodic sweeper (`@nestjs/schedule` cron every 30s) re-publishes any `notifications` row where the viewer hasn't ACKed and the row is older than 60s.
- **Idempotency on the receiver side.** The receiver dedups by `id` (already in the Android DB plan).
- **Idempotency on the bridge.** The bridge checks for the row's existence by `id` before publishing (relies on `notifications.id` being a PRIMARY KEY).
- **Reconnect on broker drop.** mqtt.js auto-reconnects with exponential backoff. On reconnect, the bridge re-subscribes and re-runs the outbox sweeper.

### What the bridge deliberately does NOT do

- Does not inspect or modify notification content (no content scanning).
- Does not enforce content-based filtering ("only SMS from these senders"). v2 could add this, but YAGNI for v1.
- Does not translate between protocols. It's a pure MQTT-to-MQTT bridge plus a DB.

## 10. Security & privacy

| Concern | Mitigation |
|---|---|
| TLS in transit | EMQX listener on 8883 (MQTTS) + 8083 (WSS). Caddy/NGINX terminates TLS in front. Cert from Let's Encrypt. |
| TLS to clients | Mandatory in prod. Self-signed allowed in dev. |
| JWT secret | HS256 with a 32+ byte secret in env. Rotate quarterly. Document rotation procedure. |
| Refresh token theft | Hashed in DB. Reuse detection: if a revoked token is presented, revoke ALL tokens for that user. |
| Device compromise | `DELETE /devices/:id` cascades to refresh tokens; the device can no longer get a new JWT; existing MQTT connections die on next reconnect (token expires in 1h, plus we can add a `revoked-clients` deny list the bridge checks on each message). |
| Spoofed source device | EMQX ACL enforces `clientid == dev_<sourceDeviceId>` matches `topic == syncwire/relay/<sourceDeviceId>`. The broker checks this on every PUBLISH. A compromised client can only publish under its own clientid. |
| PII in payloads (SMS content) | At-rest encryption in Postgres (disk-level). Application-level encryption is a v2 concern. Documented retention policy (default 30 days, configurable per user). |
| Rate limiting | `POST /api/pairings/claim`: max 5/min per user. `POST /api/notifications`: max 60/min per source device. NestJS `@nestjs/throttler`. EMQX has its own connection-rate-limit. |
| Audit trail | Every pairing creation, every pairing revocation, every login, every device revoke — written to a `audit_log` table. |
| Compliance posture | Bangladesh jurisdiction: no specific data-protection law yet but follows international norms. Add a `PRIVACY.md` covering what we collect, why, retention, and the user's right to delete (one-click "delete my account" endpoint in v1). |
| Dependency CVEs | Dependabot + weekly `npm audit` in CI. |
| Secrets in repo | `.env` is gitignored, `.env.example` is the template. CI injects real secrets. |

## 11. Edge cases — exhaustively

### On the wire

- **Duplicate publish by source.** `id` is a client-generated UUID; INSERT uses `ON CONFLICT DO NOTHING` so duplicates are silently deduped. Receiver dedups by `id` again on its side.
- **Out-of-order publish.** Bridge writes rows in arrival order, but `server_received_at` is the canonical ordering key. Receiver renders in `server_received_at` order.
- **Bridge publish fails (broker rejected).** Outbox sweeper re-publishes. After 24h, marked `dead_letter=true` and a metric fires.
- **Viewer offline when message arrives.** QoS 1 + persistent session queues at the broker. Session expiry default 2h — but the REST history endpoint catches anything older.
- **Viewer disconnects mid-message.** MQTT guarantees no partial message (PUBLISH is one packet). Receiver re-subscribes on reconnect and gets any queued.
- **Two sources, one viewer.** Each source's bridge handler publishes to the same `inbox/<viewer>` topic; receiver interleaves by `server_received_at`.
- **Clock skew on source device.** Server uses `server_received_at` for ordering; `timestamp` is preserved in the payload for the user to see. The bridge warns (metric) if `|server_received_at - timestamp| > 5 min`.
- **Very large SMS content (MMS, long messages).** Cap content at 16 KB server-side; reject longer with 413. Same cap on the publisher.

### At auth

- **JWT expires mid-session.** Client refreshes proactively (5 min before exp) using the refresh token. If it misses, on the next MQTT PUBLISH the broker drops the connection with `NotAuthorized`; the client re-authenticates and reconnects.
- **Refresh token used twice.** Reuse detection revokes the entire user's session family. Forces re-login.
- **Pairing code typed wrong.** Server returns 404 (code not found) — generic, not 400 (don't leak whether the code is well-formed but wrong).
- **Pairing code race (two devices claim at the same instant).** Postgres transaction with `SELECT ... FOR UPDATE` on the `pairing_codes` row; second one gets a 409.
- **Pairing code expired when claimed.** Returned as 410 Gone (not 404) so the client can show "code expired, generate a new one" UX.
- **JWT signature secret rotation.** Support two valid secrets (`JWT_SECRET_CURRENT`, `JWT_SECRET_PREVIOUS`) for a 24h overlap window. EMQX JWT authenticator configured with both.

### On the device (Android)

- **User revokes notification access in system Settings.** Next `NotificationListenerService.onListenerConnected` never fires; the app detects the loss and shows a "re-enable SyncWire in Settings" prompt.
- **User revokes SMS permissions (Android 13+ scoped SMS permissions).** App detects the permission loss and stops the SMS ingestion path; continues to deliver notifications; surfaces a banner.
- **Battery saver / doze.** Foreground service with `FOREGROUND_SERVICE_TYPE_SPECIAL_USE` (Android 14+) and a persistent low-priority notification. Wake lock is **not** held — we rely on MQTT keepalive (60s) and the broker queue.
- **App process killed by the OS.** WorkManager job reschedules the foreground service every 15 min as a safety net.
- **No network.** mqtt.js buffers outbound publishes (up to its in-memory queue limit). On reconnect, flushes.
- **Phone storage full.** Notification capture writes nothing to disk; everything is forwarded in-memory. Only the outbox queue might overflow, in which case we drop oldest with a `dropped_count` metric.

### Multi-user

- **User deletes account.** Cascade deletes devices, pairings, refresh tokens. `notifications` rows are anonymized: replace `sender` and `content` with `[redacted]`, keep the row shape so other users' pairing history isn't corrupted.
- **Two users pair, then one deletes account.** Pairing goes to `revoked`, surviving user's UI shows the pairing as inactive.
- **User changes email.** Re-verifies via a one-time code; refresh tokens survive (so they don't get logged out everywhere).

### Operational

- **EMQX broker crash.** Bridge auto-reconnects. Pending publishes go to the DB outbox and replay on reconnect.
- **NestJS app crash mid-publish.** Same as above — outbox sweeper.
- **DB unreachable.** Bridge stops ingesting; returns 503 on REST. Health check goes red. mqtt.js queue fills up; oldest messages are dropped past the limit. Metric fires.
- **Clock skew between server and DB.** `now()` is always on the DB. Server-side timestamps always come from the DB, not `new Date()` in the app.
- **Schema migration with running traffic.** Prisma migrations are forward-only and additive in v1. We never drop a column in the same release that removes its reads.
- **Backpressure from a slow viewer.** Broker's per-client queue has a default 1000-message cap; we set 10,000 and metric any client that hits it.

## 17. Containerization & deployment

> Added per Mohsin's direction: the whole project runs in Docker. Postgres and EMQX are not "external services" — they are siblings of the app in a single Compose stack.

### Goals

- One command (`docker compose up`) brings the entire dev stack online
- One command (`docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d`) brings prod online
- No host dependencies beyond Docker itself (no Node, no Postgres, no EMQX on the host)
- Stack is reproducible across dev machines, CI, and self-hosted prod

### Services (in every compose file)

| Service | Image | Purpose | Host port (dev) | Host port (prod) |
|---|---|---|---|---|
| `app` | built from `Dockerfile` | NestJS HTTP + MQTT bridge | 8080 (HTTP) + 9229 (debug) | 8080 (behind Caddy, internal) |
| `postgres` | `postgres:16-alpine` | Primary data store | 5432 | not exposed |
| `emqx` | `emqx/emqx:5` | MQTT broker | 1883 (MQTT) + 18083 (dashboard) | 1883 (internal), 8883 (TLS, behind Caddy) |
| `caddy` | `caddy:2-alpine` | TLS termination + reverse proxy | not present in dev | 80, 443 |

### File layout

- `Dockerfile` — multi-stage: build (npm ci + nest build) → runtime (node:20-alpine + non-root + dumb-init + HEALTHCHECK)
- `.dockerignore` — excludes `node_modules`, `.git`, `.plan`, `.progress`, `dist`, `coverage`, `test`, `*.log`, `Dockerfile`, `.dockerignore`
- `docker-compose.yml` — base stack (app + postgres + emqx + named volumes)
- `docker-compose.override.yml` — dev only: bind-mount `./src` and `./prisma` for hot-reload, expose Node debug port 9229, bind-mount `./deploy/*` read-only
- `docker-compose.prod.yml` — prod only: no debug port, no bind-mounts, `restart: always`, Caddy service for TLS, `env_file: .env.prod`, healthcheck intervals tightened
- `deploy/postgres/init.sql` — `CREATE EXTENSION IF NOT EXISTS pgcrypto;` (for `gen_random_uuid()`), creates the `syncwire` role
- `deploy/emqx/emqx.conf` — listeners, log level, dashboard port, max connections
- `deploy/emqx/acl.conf` — topic ACL rules with placeholders (see §5)
- `deploy/emqx/authn-jwt.conf` — JWT authenticator chain (HS256, secret from env, claim-to-ACL mapping)
- `deploy/caddy/Caddyfile` — automatic HTTPS via Let's Encrypt, reverse-proxy `/api/*` to `app:8080`, reverse-proxy MQTT-over-WebSocket to `emqx:8083`

### Networks

- A single user-defined bridge network `syncwire-net`
- All services on `syncwire-net`
- Only `caddy` and (optionally) the `emqx` dashboard publish host ports in prod
- The `app` and `emqx` talk to each other by service name — no host networking

### Volumes

| Volume | Mounted in | Purpose |
|---|---|---|
| `pg_data` | `/var/lib/postgresql/data` in `postgres` | durable DB |
| `emqx_data` | `/opt/emqx/data` in `emqx` | durable broker state (retained messages, sessions) |
| `emqx_log` | `/opt/emqx/log` in `emqx` | log persistence for debugging |
| `caddy_data` | `/data` in `caddy` | TLS certs + OCSP staples |
| `caddy_config` | `/config` in `caddy` | caddy runtime config |

No host bind-mounts in prod. In dev, only `app`'s `./src` and `./prisma` are bind-mounted (for hot-reload + live Prisma client regen); config files in `deploy/*` are bind-mounted read-only so changes take effect on container restart.

### Health & startup ordering

- `postgres` healthcheck: `pg_isready -U $POSTGRES_USER`
- `emqx` healthcheck: `curl -f http://localhost:18083/api/v5/health || exit 1`
- `app` healthcheck: `wget -qO- http://localhost:8080/api/health || exit 1` (NestJS `/health` checks DB + EMQX connection)
- `app` uses `depends_on: { postgres: { condition: service_healthy }, emqx: { condition: service_healthy } }` so it doesn't start until both backends are ready
- Prisma migrations run as a one-shot init container (`app-migrate`) before `app` starts in prod; in dev, `app` runs `prisma migrate deploy` itself on boot via the entrypoint

### App image build

- Multi-stage: builder (Node 20, full deps) → runtime (Node 20-alpine, prod deps only, non-root user, dumb-init)
- Built for `linux/amd64` and `linux/arm64` (so it runs on a Raspberry Pi as well as x86 servers)
- Tagged with both `latest` and the git SHA (`ghcr.io/mohsin/syncwire-server:<sha>`)
- Pushed to GitHub Container Registry by `.github/workflows/docker.yml` on tags
- Image size target: under 200 MB

### Environment & secrets

- Dev: `.env` at the project root, mounted into the `app` and `emqx` services via `env_file`
- Prod: `.env.prod` (gitignored) on the host, same mechanism. Alternatively Docker Swarm secrets or `secrets:` in Compose for higher-security environments — keep as a v1.1 upgrade path
- The same `.env.example` documents every variable for both dev and prod

### Backup & restore

- Postgres: nightly `pg_dump` to a host-mounted volume (`./backups/postgres`) via a `backup` service with a `cron`-like schedule, or as a host cron job — implementation choice in phase 6 hardening
- EMQX data: durable, but can be wiped (no durable state required — retained messages are best-effort, sessions are best-effort, persistent state lives in Postgres)
- Restore procedure documented in the runbook (phase 6)

### Local dev loop

```bash
# bring up the full stack with hot-reload
docker compose up

# in another shell, run prisma migrations once after schema changes
docker compose exec app npx prisma migrate dev

# run tests inside the app container
docker compose exec app npm test

# tail logs for a service
docker compose logs -f app

# tear down (keeps volumes)
docker compose down

# tear down including volumes (nuke)
docker compose down -v
```

### Prod loop

```bash
# on the host, first time only
cp .env.example .env.prod && edit

# deploy
docker compose -f docker-compose.yml -f docker-compose.prod.yml pull
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

# rolling restart of just the app after a code change
docker compose -f docker-compose.yml -f docker-compose.prod.yml pull app
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d app
```

### Scaling (when we outgrow a single host)

- Postgres: switch to a managed service (RDS / Cloud SQL / Neon / Supabase) and remove the `postgres` service from Compose
- EMQX: run as a 3-node EMQX cluster behind a TCP load balancer (HAProxy or cloud LB); the app connects to the LB
- App: scale horizontally — stateless, behind a load balancer
- Volumes become external (managed Postgres, EMQX cluster storage)
